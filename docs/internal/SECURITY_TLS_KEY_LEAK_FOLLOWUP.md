# Follow-up: потенциальная утечка TLS private key через git (internal)

> **Статус:** runbook оценки и реагирования. Не заменяет triplet `PROD_DEPLOY` / `DATA_MIGRATION` / `RELEASE_CHECKLIST`.  
> **Контекст:** в старом `SSL_SETUP_INSTRUCTIONS.md` обнаруживались PEM (в т.ч. `BEGIN RSA PRIVATE KEY`) и host-like детали. Файл снят с рабочего дерева, добавлен [SSL_TLS_SERVER_TEMPLATE.md](SSL_TLS_SERVER_TEMPLATE.md). **История git** может по-прежнему содержать старые объекты — см. раздел «История» ниже.

## 1. Факт, если private key в истории

- Любой, кто **когда-либо** имел доступ к **репозиторию** (clone, fork, бэкап, CI-артефакты, старые ноутбуки), **теоретически** мог извлечь ключ из истории, даже после удаления файла в текущем `HEAD`.  
- Это **не** «само по себе взлом сервера» — но при **совпадении** с реальным production key пары `cert`/`key` на хосте злоумышленник с копией ключа может выдавать **валидные** соединения для того же сертификата (пока серт не отозван) и расшифровывать прошлый/текущий трафик к MITM в зависимости от контекста; на практике критично: **владение тем же private key, что и nginx**.

## 2. Severity (сдержанно)

- **Высокая**, если leaked key **идентичен** текущему `ssl_certificate_key` на prod **или** сертификат ещё **не** ротирован и key не отозван.  
- **Средняя/низкая по impact**, если доказано, что материал был **тестовым/устаревшим** и **никогда** не кладутся пути, которые сейчас в nginx.  
- **Неполная уверенность** → обрабатывать как **C** (см. decision tree): сначала проверка, потом ротация при малейшем сомнении.

## 3. Безопасный verification checklist

Выполнять на **сервере** (SSH) и/или **локально** с копий, **без** вставки секретов в тикеты/чат.

### 3.1 Nginx: какие пути к файлам

- `sudo nginx -T 2>/dev/null | rg ssl_certificate` или просмотр `sites-enabled` / `conf.d` для `dedato.ru` (или актуального `server_name`).  
- Записать **только пути** к `ssl_certificate` и `ssl_certificate_key` (не содержимое в тикет).

### 3.2 Сопоставление **публичного** cert без утечки PEM

- **С сервера (файл):**  
  `openssl x509 -in /path/to/cert_or_fullchain -noout -fingerprint -sha256`  
  `openssl x509 -in ... -noout -serial -subject -issuer -dates`  
- **С домена (runtime):**  
  `openssl s_client -connect dedato.ru:443 -servername dedato.ru </dev/null 2>/dev/null | openssl x509 -noout -fingerprint -sha256 -serial -dates`  
- Сравнить **fingerprint (SHA-256)** и **serial** между «что в nginx» и «что в ответе :443». Несоответствие → разобрать конфиг/несколько vhost.

### 3.3 Сопоставление **private key** с cert (без публикации ключа)

- На сервере, **локально** (stdout — только **hash**, не ключ):  
  `openssl pkey -in /path/to/key -pubout | openssl sha256`  
  `openssl x509 -in /path/to/cert_or_fullchain -noout -pubkey | openssl sha256`  
  Должны **совпасть** шестнадцатеричные хеши **публичного** ключа, извлечённого из key и из cert.  
- **Сравнение с утечкой:** если у вас есть **офлайн** копия старого ключа (например, из бэкапа расследования) — **тот же** pipeline `pkey -pubout | sha256` и сравнить с **текущим** `/path/to/key` на сервере. Совпадение → тот же ключ. **Не** копируйте ключ в репо.

### 3.4 Modulus (альтернатива)

- `openssl rsa -in key -noout -modulus | openssl sha256`  
- `openssl x509 -in cert -noout -modulus | openssl sha256`  
  Должны совпасть. Для сравнения **двух** ключей (старый leak vs текущий файл) — сравнить **только** хеши модулей, **не** сами модулы в публичные каналы.

### 3.5 Что ещё проверить

- **Срок действия** cert (`notAfter`) — не путать «новый LE cert» с «новый key» (LE часто тот же key).  
- **mtime** файлов key/cert на диске (оринентир, не доказательство).  
- **revocation:** для публичного CA смотрите, доступен ли OCSP/CRL; для Let’s Encrypt обычно **ротация = новый cert + новый key**.

## 4. Decision tree (кратко)

| Сценарий | Действие |
|----------|----------|
| **A.** Проверка показала: текущий prod key **==** материалу из утечки (или невозможно отличить) | **Срочно:** ротация key + перевыпуск/установка cert, перезагрузка nginx, проверка :443. Параллельно: инцидент-запись, кто имел git-доступ. |
| **B.** Доказано: на сервере **другой** key pair, старый **не** используется; leak был тест/черновик | **Всё равно:** убедиться, что **история** git и бэкапы обработаны (см. §6), обновить политику; ротация на prod **опциональна** по риску, но снятие секрета из history — по политике. |
| **C.** **Нет уверенности** | **Сначала** checksum/fingerprint (§3), **не** паниковать. Если проверка затруднена — **закрыть риск ротацией** + новый key (часто дешевле, чем расследование). |
| **D.** Репозиторий **приватный**, мало клонов | **Риск снижен**, инцидент **не** отменён: инсайдеры, бывшие сотрудники, скомпрометированный токен, зеркала, бэкапы. Действия **те же по логике**, возможно **мягче по срочности** ротации **после** доказательства «key не тот». |

## 5. Порядок действий (рекомендуемый)

1. **Зафиксировать** факт: дата обнаружения, коммит(ы) (после `git log` по пути старого файла — **без** вывода секретов).  
2. **Проверка** по §3 (fingerprint, key–cert match, runtime vs file).  
3. Если **A** или **C+ doubt:** **ротация** key+cert на prod, smoke-test HTTPS, мониторинг ошибок.  
4. **Документация** инцидента (см. §7 в ответе пользователю / внутренний Confluence).  
5. **История git** (§6) — **после** ротации или **параллельно** при отдельной ветке/форке; не блокировать смену ключа ожиданием rewrite.

## 6. Git history: когда чистить

- **Достаточно удаления в `HEAD`:** если риск приемлем (тестовый ключ, repo никогда не публиковался, мало копий) — **иногда** достаточно + ротация на всякий.  
- **Недостаточно:** если key **мог** быть production и repo **клонировался** широко, есть **public** зеркала, **CI** с полным fetch, **форки**.  
- **filter-repo / BFG:** оправдано, если политика **«секретов в истории не должно остаться»** и готовы **force-push** + пересинхрон всех. **Риски:** сбой у разработчиков, старые PR, клоны «вне» орг-контроля **всё ещё** хранят старые объекты.  
- **Безопасный порядок:** **сначала** ротировать **реальные** ключи на сервере (закрыть вектор), **затем** планировать rewrite + уведомить команду. Ротация **не** отменяет desire purge history, но **снижает** оставшийся risk.

## 7. Документирование (без паники и без замалчивания)

- **Внутренний** incident log: что было, что проверили, итог fingerprint, ротировали или нет, **без** PEM.  
- **Post-mortem (короткий):** root cause = commit secret; preventive = pre-commit hook / secret scan (gitleaks), запрет копипаста ключей, review.  
- Публично — **только** если есть обязанность (утечка PII / регулятор); иначе **внутренняя** фиксация.
