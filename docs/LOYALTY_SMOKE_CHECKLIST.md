# Smoke: лояльность и скидки (клиент / публичная запись)

Компактный ручной прогон по **реально исполняемым** правилам рантайма. Источник логики: `backend/utils/loyalty_discounts.py` (`evaluate_discount_candidates`, `evaluate_and_prepare_applied_discount`), публичная запись: `backend/routers/public_master.py`, клиентские брони: `backend/routers/client.py`, общие брони: `backend/routers/bookings.py`.

**Регрессия 500 на `GET .../booking-price-preview`:** при `start_time` с offset (`...+03:00`) `booking_start` приводится к **naive UTC**. Если `last_visit` из БД (PostgreSQL) приходит как **timezone-aware**, выражение `(naive_booking_start - aware_last_visit)` давало `TypeError` — в ветке `returning_client` `last_visit` тоже приводится к naive UTC. При ошибке preview в лог пишется `booking-price-preview failed …` (`public_master.get_booking_price_preview`, `logger.exception`). Публичные ответы **не показывают** `[QA SMOKE]` в `rule_name` (сид в БД с маркером сохраняется).

## 1. Карта правил (аудит)

| Правило (condition_type / тип) | Где считается (SoT) | Публичная запись `/m/:slug` | До выбора услуги (UI) | После выбора услуги + слота (UI) | Только цена / бейдж |
|--------------------------------|---------------------|-----------------------------|------------------------|-------------------------------------|----------------------|
| **birthday** | `evaluate_*` → `POST /api/public/masters/{slug}/bookings`, `GET .../booking-price-preview`, ранний hint в `GET .../eligibility` (`loyalty_hint`) | Да | Баннер `data-testid="public-loyalty-hint-banner"` если `eligibility.loyalty_hint.active` (опорная дата = сегодня 12:00 в TZ мастера, без услуги) | Блок сводки: база / скидка / к оплате из **preview** | Оба: бейдж рано, **итоговая сумма** после слота |
| **first_visit** | То же | Да | Может попасть в `loyalty_hint` без услуги | Preview + POST | То же |
| **returning_client** | То же (нужны `booking_start` + история визитов) | Да | Обычно **нет** в раннем hint (нет даты визита в смысле слота; в коде hint всё же с «сегодня» — может врать относительно финального слота) | Preview/POST по **реальному** `start_time` | Цена; ранний бейдж — ориентир |
| **regular_visits** | То же (окно визитов от «сейчас» мастера) | Да | Может быть в hint | Preview/POST | То же |
| **happy_hours** | То же (день недели + интервал по **времени визита**) | Да | **Ненадёжно** в hint (опора полдень сегодня) | Preview/POST по слоту — **SoT** | В основном цена после слота |
| **service_discount** | То же (привязка к `service_id` / категории канонического `Service`) | Да | **Нет** в hint (`service_id=null` в eligibility) | Preview/POST после выбора услуги | Цена + breakdown |
| **personal** (`PersonalDiscount`, телефон) | То же, отдельная ветка в `evaluate_discount_candidates` | Да | Может быть в hint без услуги | Preview/POST | Бейдж + цена |
| **Баллы лояльности** (списание) | Другие эндпоинты / формы (напр. `MasterBookingModule`, ЛК) | **Не** этот чеклист публичной скидки %-ми | — | — | Отдельный сценарий |
| **Промокод / complex enum** (`promo_code`, `visit_count`, … в `LoyaltyConditionType`) | В **runtime** скидок **не** участвуют (`SUPPORTED_CONDITION_TYPES` в `loyalty_discounts.py`; создание quick через API отклонит неизвестный `condition_type`) | Нет | — | — | Не smoke-ить как %-скидку записи |

Кратко: см. раздел **«Что считать source of truth при smoke-проверке»** ниже.

## 2. Список типов в продукте (код)

**Участвуют в расчёте скидки записи:** `birthday`, `first_visit`, `returning_client`, `regular_visits`, `happy_hours`, `service_discount`, персональная скидка (`PersonalDiscount`).

**Есть в enum модели, но не в рантайме quick-скидок записи:** `promo_code`, `visit_count`, `spent_amount`, и пр. из `models.LoyaltyConditionType` — не использовать в smoke %-скидки до появления поддержки в `SUPPORTED_CONDITION_TYPES` и `evaluate_discount_candidates`.

## Проверка приоритета и конфликта правил

**Как выбирается одно правило, если матчатся несколько**

1. Сначала среди матчей берётся **максимальный `discount_percent`**.
2. Если **проценты равны**, смотрится приоритет **`condition_type`** (в коде: `birthday` выше `first_visit`, персональная ниже quick — см. `CONDITION_TYPE_PRIORITY` в `loyalty_discounts.py`).
3. Дальше — детерминированный tie-break по `rule_id`.

**Ручные сценария проверки (коротко)**

| # | Что сделать | Что ожидать в preview |
|---|-------------|------------------------|
| A | **`+79991500101`**: 0 completed + ДР в окне + `first_visit` 10% и `birthday` 12%, слот в окне ДР | **12%** (max %) |
| B | Два правила с **одинаковым** % (напр. оба 12%: `birthday` и personal для разных веток) — см. настройки | При равном % побеждает тип с **лучшим** приоритетом (для smoke чаще **birthday** vs personal) |
| C | **`+79991500102`** + «Стрижка мужская»: `first_visit` 10% vs `service_discount` 9% | **10%** (10 > 9) |

Не нужно читать код во время smoke: достаточно сверить **preview** с ожидаемым % из таблицы выше.

## Что считать source of truth при smoke-проверке

| Уровень | Что это | Как использовать |
|---------|---------|-------------------|
| Ранний баннер (`eligibility.loyalty_hint`, `public-loyalty-hint-banner`) | **Предпросмотр** по опорной дате (сегодня 12:00, TZ мастера) **без** выбранной услуги | Быстро понять «есть ли шанс на скидку»; расхождение со слотом — **не всегда баг** |
| `GET .../booking-price-preview` | Тот же расчёт, что перед `POST` (для выбранных услуги и `start_time`) | **Главный ориентир** для суммы в UI перед отправкой формы |
| `POST /api/public/masters/{slug}/bookings` + ответ / запись в БД | Окончательное применение | Если preview и POST расходятся — искать гонку слота, смену правил, ошибку сети |

Если баннер говорит «есть скидка», а preview — нет: проверить **тип правила** (ДР vs happy hours vs service_discount), **дату/время слота**, **услугу**, **completed-историю**.

## 4. Чеклист по правилам (детально)

Общие шаги: открыть `/m/<domain>`, при необходимости залогиниться клиентом с нужным телефоном/ДР/историей.

### 4.1 День рождения (`birthday`)

| Шаг | Действие |
|-----|----------|
| Предусловия | У клиента в профиле `birth_date`; у мастера активное правило `birthday` с `days_before` / `days_after`; дата визита попадает в окно. |
| Данные | Smoke: правило `[QA SMOKE] birthday`; клиент **`+79991500101`** — `birth_date` выставляется reseed’ом так, что **сегодня** (дата прогона) в TZ мастера попадает в окно ±7. |
| Открыть | `/m/qa-smoke-public` (или ваш domain). |
| До услуги | Авторизованный клиент: зелёный баннер `public-loyalty-hint-banner` (если hint активен). Гость: баннера нет. |
| После услуги + слота | В сводке строки база / скидка / к оплате; совпадает с ожидаемым %. |
| Итог | `POST .../bookings` создаёт запись с `payment_amount` со скидкой; в БД `applied_discounts` при наличии правила. |

### 4.2 Первый визит (`first_visit`)

| Шаг | Действие |
|-----|----------|
| Предусловия | Нет **completed** визитов у этого `client_id` к этому `master_id`. |
| Данные | Smoke: **`+79991500102`** (0 completed к M8); для чистого **10%** без SD — услуга **«Стрижка женская»**. |
| До услуги | Возможен баннер при активном правиле. |
| После слота | Preview и итог при первом визите. |
| Негатив | После одного completed то же правило не матчится — скидки нет (проверить другим клиентом с историей). |

### 4.3 Повторный визит (`returning_client`)

| Шаг | Действие |
|-----|----------|
| Предусловия | Есть последний completed; `booking_start` в пределах min/max дней от последнего визита (см. `loyalty_params`). |
| Данные | Smoke: **`+79991500103`**; 1× completed **35** дней назад; слот в окне от last visit (≥30 дн.). |
| До услуги | Баннер может не отражать финал — ориентир. |
| После слота | Preview — SoT. |

### 4.4 Регулярные визиты (`regular_visits`)

| Шаг | Действие |
|-----|----------|
| Предусловия | За `period_days` не меньше `visits_count` completed к мастеру. |
| Данные | Smoke: **`+79991500104`**; 2× completed в пределах 60 дней. |
| До/после | Как выше; финал после слота. |

### 4.5 Счастливые часы (`happy_hours`)

| Шаг | Действие |
|-----|----------|
| Предусловия | День недели и время слота попадают в `parameters.days` + `intervals`. |
| Данные | Reseed: основное **happy_hours** активно (вт 09–12); второе правило — **inactive-clone** (`is_active=false`, маркер `HH_INACTIVE_CLONE` в `description`). **`+79991500109`**: 0 completed — во вт 09–12 **13%**. **`+79991500105`**: 1 completed — слот **вне** вт 09–12 (напр. среда) → **0%**. |
| До услуги | Не полагаться на баннер. |
| После слота | Выбрать слот внутри интервала → preview показывает скидку. |

### 4.6 Скидка на услугу (`service_discount`)

| Шаг | Действие |
|-----|----------|
| Предусловия | В правиле указан `service_id` (канонический id услуги мастера в параметрах — см. создание через API в smoke). |
| До услуги | Баннера по этому правилу **нет**. |
| После | Только если выбрана нужная услуга — preview и POST. |

### 4.7 Персональная скидка (`PersonalDiscount`)

| Шаг | Действие |
|-----|----------|
| Предусловия | Запись в `personal_discounts` на телефон клиента, мастер тот же. |
| Данные | Smoke: **`+79991500107`** / `test123` (§3a; персоналка только на этот телефон). |
| До услуги | Возможен баннер. |
| После слота | Preview + итог. |

## 6. Что сознательно не в этом smoke

- **Списание баллов** в другом UI (не `PublicBookingWizard`).  
- **Промокоды как `LoyaltyConditionType.PROMO_CODE`** — нет в `evaluate_discount_candidates`.  
- **Сложные (complex) правила** из enum без поддержки в `SUPPORTED_CONDITION_TYPES`.
