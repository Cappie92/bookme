# Backend uvicorn: причина ModuleNotFoundError: No module named 'src'

## Причина (что изменилось и почему сломалось)

- В репозитории **нет пакета `src`** в backend: нет каталога `backend/src/`, нет `backend/src/app/main.py`. Точка входа — **`backend/main.py`** (модуль `main`, объект `app`).
- Запуск вида `uvicorn src.app.main:app` предполагает структуру `backend/src/app/main.py` и пакет `src`. При таком импорте Python ищет модуль `src` и падает с `ModuleNotFoundError: No module named 'src'`.
- С `--reload` uvicorn порождает дочерний процесс (SpawnProcess), который заново выполняет импорт переданного модуля. В этом процессе тоже нет пакета `src` (и PYTHONPATH тут ни при чём — в репо просто нет такой структуры). Поэтому ошибка проявляется в reload subprocess.
- Раньше везде в репо использовалась команда **`main:app`** из каталога `backend/` (README, Makefile, скрипты, доки). Ошибка возникла из‑за использования команды с **`src.app.main:app`**, которая не соответствует фактической структуре проекта.

## Что именно поменять (файлы + суть правок)

### 1. Не использовать `src.app.main`

- **Команда запуска:** использовать **`main:app`**, а не `src.app.main:app`.
- **Рабочий каталог:** запуск из **`backend/`** (`cd backend`), без обязательного `PYTHONPATH`.

### 2. Внесённые правки в репо

- **backend/main.py** — в начало файла добавлен комментарий:
  - «Точка входа uvicorn: main:app (запуск из каталога backend/). Пакета src/ в проекте нет — не использовать src.app.main.»
- **README.md** — в блок «локальная разработка» добавлено:
  - Пояснение, что точка входа — `main:app` из `backend/`, пакета `src/` нет.
  - В команду uvicorn добавлены `--host 0.0.0.0 --port 8000` для явного указания порта и доступа с других устройств (в т.ч. для Vite proxy).

### 3. Временная диагностика (по желанию)

Если нужно проверить CWD/PYTHONPATH в основном и в дочернем (reload) процессе, в **backend/main.py** перед созданием FastAPI-приложения можно временно добавить:

```python
import os, sys
print("CWD", os.getcwd())
print("PYTHONPATH", os.environ.get("PYTHONPATH"))
print("sys.path[0:5]", sys.path[:5])
```

После проверки этот блок удалить. При правильной команде (`main:app` из `backend/`) CWD будет `.../backend`, а импорт `src` не выполняется, поэтому проблема с `src` не зависит от PYTHONPATH.

## Команда запуска dev (одна строка)

Из корня репозитория:

```bash
cd backend && python3 -m uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

Почему это стабильно:

- Запуск из `backend/` — текущий каталог в `sys.path`, модуль `main` находится как `backend/main.py`.
- Используется реально существующий модуль `main` и объект `app` в нём.
- Не нужен `PYTHONPATH` и не используется несуществующий пакет `src`.
- Reload-процесс наследует тот же рабочий каталог и снова импортирует `main:app` без обращения к `src`.

## Команда проверки

```bash
curl -s http://localhost:8000/health
```

Ожидаемый ответ — JSON с полем `status` (или аналог), либо 200 OK от эндпоинта `/health`. Если эндпоинта нет — проверять корень или docs:

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/
```

## Два «железных» способа запуска (на выбор)

**A) Без PYTHONPATH (рекомендуется, уже используется в репо)**  
Запуск из каталога backend с модулем `main:app`:

```bash
cd backend && python3 -m uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

**B) Через установку пакета**  
Сейчас в backend есть только `pyproject.toml` с инструментами (black, isort, pytest), без объявления пакета. Чтобы запускать из любого каталога, нужно добавить в backend минимальный `pyproject.toml` с `[project]` и опцией установки в режиме разработки (`pip install -e .`) и тогда импортировать приложение как установленный пакет. Для текущей плоской структуры (все файлы в backend/) способ A проще и уже зафиксирован в README.
