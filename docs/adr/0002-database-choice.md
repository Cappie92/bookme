# ADR-0002: Выбор базы данных

**Дата:** 2024-10-21

**Статус:** Принято

**Контекст:** Команда разработки DeDato

---

## Контекст и проблема

Для проекта DeDato необходимо выбрать систему управления базой данных, которая будет хранить:

- Пользователей (клиенты, мастера, владельцы салонов, администраторы)
- Бронирования и расписания
- Услуги и прайс-листы
- Финансовые данные (доходы, расходы, налоги)
- Настройки и конфигурации

### Требования

**Функциональные:**
- ACID транзакции для критичных операций (бронирование, оплата)
- Поддержка сложных связей между таблицами
- Полнотекстовый поиск (по услугам, салонам)
- Миграции схемы

**Нефункциональные:**
- Простота развертывания для разработки
- Минимальные требования к инфраструктуре на старте
- Возможность масштабирования в будущем
- Совместимость с SQLAlchemy ORM

**Ограничения:**
- Бюджет на инфраструктуру ограничен на старте
- Небольшая команда разработки (нет выделенного DBA)
- Развертывание в Docker

## Рассмотренные варианты

### Вариант 1: SQLite

**Описание:** Встроенная реляционная база данных, файл-ориентированная

**Плюсы:**
- Нулевая конфигурация - работает из коробки
- Не требует отдельного сервера БД
- Отличная производительность для чтения
- Простое резервное копирование (один файл)
- Идеально для разработки и тестирования
- Поддержка ACID транзакций
- Минимальные требования к памяти

**Минусы:**
- Ограниченная конкурентность записи (один писатель)
- Максимальный размер БД ~281 TB (достаточно, но есть лимит)
- Нет встроенной репликации
- Ограниченные возможности для масштабирования
- Не подходит для высоких нагрузок записи

### Вариант 2: PostgreSQL

**Описание:** Мощная open-source реляционная СУБД

**Плюсы:**
- Отличная конкурентность (MVCC)
- Мощные возможности индексирования
- Полнотекстовый поиск
- JSON/JSONB поддержка
- Репликация и масштабирование
- Большое сообщество

**Минусы:**
- Требует отдельного сервера
- Сложнее настройка и обслуживание
- Выше требования к ресурсам
- Оверкилл для начального этапа проекта

### Вариант 3: MySQL/MariaDB

**Описание:** Популярная open-source реляционная СУБД

**Плюсы:**
- Широкое распространение
- Хорошая производительность
- Репликация

**Минусы:**
- Аналогичные PostgreSQL недостатки для начального этапа
- Менее строгая ACID поддержка в некоторых движках
- Меньше advanced features чем PostgreSQL

## Принятое решение

**Выбран:** Вариант 1 (SQLite) для начального этапа с планом миграции на PostgreSQL

### Обоснование

**Стратегия:** Использовать SQLite для разработки и раннего production, мигрировать на PostgreSQL при росте нагрузки.

**Почему SQLite сейчас:**

1. **Простота развертывания:** Нулевая конфигурация, нет зависимостей
2. **Скорость разработки:** Мгновенное начало работы, легкое тестирование
3. **Производительность:** Для текущей нагрузки (<100 пользователей) SQLite быстрее
4. **Резервное копирование:** Простое - копирование одного файла
5. **Стоимость:** Нет затрат на инфраструктуру БД

**Когда мигрировать на PostgreSQL:**

- >1000 активных пользователей
- >100 одновременных записей
- Необходимость репликации
- Размер БД >10GB
- Необходимость advanced features (полнотекстовый поиск, JSONB)

**Гарантия миграции:**

- SQLAlchemy ORM абстрагирует базу данных
- Alembic миграции совместимы с обеими БД
- Тестирование с обеими БД в CI/CD

## Последствия

### Положительные

- Быстрый старт проекта без настройки БД
- Простое развертывание в Docker (один контейнер)
- Легкое резервное копирование и восстановление
- Отличная производительность для текущей нагрузки
- Нет затрат на инфраструктуру БД
- Простота тестирования (in-memory SQLite для unit-тестов)

### Отрицательные

- Ограничения конкурентности при росте нагрузки
- Необходимость планировать миграцию заранее
- Невозможность горизонтального масштабирования
- Отсутствие некоторых advanced features

### Риски

**Риск 1:** Нагрузка вырастет раньше чем ожидалось

**Митигация:**
- Мониторинг производительности БД
- Готовые миграционные скрипты
- Тестирование с PostgreSQL в staging среде

**Риск 2:** Сложности при миграции с SQLite на PostgreSQL

**Митигация:**
- Использование SQLAlchemy ORM (абстракция БД)
- Избегание SQLite-специфичных запросов
- Регулярное тестирование с PostgreSQL
- Документирование различий между БД

**Риск 3:** Потеря данных при сбое файловой системы

**Митигация:**
- Автоматические ежедневные бэкапы
- Репликация файла БД на резервный сервер
- WAL (Write-Ahead Logging) включен

## Детали реализации

### Конфигурация SQLite

```python
# backend/database.py
SQLALCHEMY_DATABASE_URL = "sqlite:///./bookme.db"

engine = create_engine(
    SQLALCHEMY_DATABASE_URL,
    connect_args={
        "check_same_thread": False,  # Для FastAPI
        "timeout": 30                 # Таймаут для блокировок
    },
    poolclass=StaticPool,             # Для SQLite
)

# Включение WAL mode для лучшей конкурентности
@event.listens_for(Engine, "connect")
def set_sqlite_pragma(dbapi_conn, connection_record):
    cursor = dbapi_conn.cursor()
    cursor.execute("PRAGMA journal_mode=WAL")
    cursor.execute("PRAGMA synchronous=NORMAL")
    cursor.execute("PRAGMA foreign_keys=ON")
    cursor.close()
```

### Миграции (Alembic)

```ini
# alembic.ini
[alembic]
script_location = alembic
sqlalchemy.url = sqlite:///./bookme.db
```

### Backup Strategy

```bash
# Ежедневный бэкап
#!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
cp bookme.db backups/bookme_backup_$DATE.db
# Хранить бэкапы за последние 30 дней
find backups/ -name "bookme_backup_*.db" -mtime +30 -delete
```

### Мониторинг

```python
# Метрики для мониторинга необходимости миграции
- Количество одновременных подключений
- Время выполнения запросов
- Размер файла БД
- Количество блокировок записи
```

### План миграции на PostgreSQL

```yaml
Этап 1: Подготовка
  - Настроить PostgreSQL сервер
  - Создать схему через Alembic
  - Протестировать все запросы

Этап 2: Миграция данных
  - Экспорт данных из SQLite
  - Импорт в PostgreSQL
  - Валидация данных

Этап 3: Переключение
  - Режим read-only для SQLite
  - Финальная синхронизация
  - Переключение connection string
  - Тестирование

Этап 4: Мониторинг
  - Проверка производительности
  - Откат если необходимо
```
