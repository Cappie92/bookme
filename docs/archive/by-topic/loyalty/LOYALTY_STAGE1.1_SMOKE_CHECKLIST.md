## Быстрые скидки (7 проверок)

### ✅ 1. Открытие таба "Быстрые"
**Действие:** Открыть экран "Лояльность" → таб "Скидки" → подтаб "Быстрые"  
**Ожидаемо:**
- Загружаются шаблоны (GET /api/loyalty/templates)
- Загружаются активные скидки (GET /api/loyalty/status)
- Сетка шаблонов отображается с иконками, названиями, описаниями
- Список активных скидок отображается (если есть)

### ✅ 2. Создание скидки из шаблона (кнопка "Активировать")
**Действие:** Нажать "Активировать" на шаблоне (например, "Новый клиент")  
**Ожидаемо:**
- POST /api/loyalty/quick-discounts отправляется с правильным body
- После успеха → GET /api/loyalty/status (refetch)
- Шаблон помечается как "Активна" (зелёный бейдж)
- Кнопка "Активировать" становится disabled "Активна"
- Скидка появляется в списке "Активные быстрые скидки"

### ✅ 3. Создание скидки с кастомным процентом
**Действие:** Нажать ✏️ на шаблоне → ввести процент (например, 25) → "Сохранить"  
**Ожидаемо:**
- POST /api/loyalty/quick-discounts отправляется с customPercent=25
- Скидка создаётся с процентом 25% (не default_discount)
- Шаблон помечается как "Активна"

### ✅ 4. Редактирование процента активной скидки
**Действие:** Нажать ✏️ на активной скидке → ввести новый процент → "Сохранить"  
**Ожидаемо:**
- PUT /api/loyalty/quick-discounts/{id} отправляется с { discount_percent: <новый> }
- После успеха → GET /api/loyalty/status (refetch)
- Процент обновляется в UI
- Режим редактирования закрывается

### ✅ 5. Удаление быстрой скидки
**Действие:** Нажать 🗑️ на активной скидке → подтвердить удаление  
**Ожидаемо:**
- Alert.alert с confirm "Вы уверены, что хотите удалить эту скидку?"
- При подтверждении → DELETE /api/loyalty/quick-discounts/{id}
- После успеха → GET /api/loyalty/status (refetch)
- Скидка исчезает из списка
- Шаблон больше не "Активна"

### ✅ 6. Граничные значения процента
**Действие:** Попытаться ввести 0, 1, 100, 101 в поле процента  
**Ожидаемо:**
- 0, 1, 100 → принимаются (валидация проходит)
- 101 → Alert.alert "Процент должен быть от 0 до 100" (или backend валидация)

### ✅ 7. Шаблоны 404
**Действие:** Если GET /api/loyalty/templates возвращает 404  
**Ожидаемо:**
- Ошибка игнорируется (не показывается Alert)
- setTemplates([]) → пустая сетка шаблонов
- Нет краша приложения

## Обработка ошибок (5 проверок)

### ✅ 19. 400 Validation Error
**Действие:** Отправить невалидные данные (например, discount_percent > 100)  
**Ожидаемо:**
- Alert.alert с деталью из error.response?.data?.detail
- UI не крашится

### ✅ 20. 401 Unauthorized (до authReady)
**Действие:** Открыть экран до загрузки auth (isLoading = true)  
**Ожидаемо:**
- Запросы НЕ отправляются (auth gating работает)
- Нет 401 ошибок в логах
- После authReady → запросы отправляются

### ✅ 21. 403 Forbidden (SUBSCRIPTION_REQUIRED)
**Действие:** Открыть экран с тарифом без доступа к лояльности  
**Ожидаемо:**
- GET /api/loyalty/status возвращает 403 с X-Error-Code: SUBSCRIPTION_REQUIRED
- Показывается жёлтый блок (warningCard) с текстом "Доступ к разделу «Лояльность» доступен в подписке..."
- Кнопка "Управление подпиской" ведёт на /subscriptions

### ✅ 22. 404 Master not found
**Действие:** Открыть экран с пользователем без профиля мастера  
**Ожидаемо:**
- GET /api/loyalty/status возвращает 404
- Показывается красный блок (errorCard) с текстом "Профиль мастера не найден. Пожалуйста, перелогиньтесь..."
- Списки скидок пустые

### ✅ 23. 409 SCHEMA_OUTDATED
**Действие:** Открыть экран с устаревшей схемой БД (миграция не применена)  
**Ожидаемо:**
- GET /api/loyalty/status возвращает 409 с X-Error-Code: SCHEMA_OUTDATED
- Показывается жёлтый блок (warningCard) с текстом "Loyalty schema outdated, apply migrations. Run alembic upgrade head"
- UI не крашится

## Edge cases (2 проверки)

### ✅ 24. Пустые списки (нет скидок)
**Действие:** Открыть таб, где нет скидок (новый мастер)  
**Ожидаемо:**
- Для быстрых: показывается "Нет быстрых скидок" (если нет шаблонов и скидок)
- Для сложных: показывается только кнопка "Создать сложную скидку"
- Для персональных: показывается "Персональные скидки не настроены" (если форма не открыта)

### ✅ 25. Одновременное редактирование (несколько пользователей)
**Действие:** Два мастера одновременно редактируют одну скидку  
**Ожидаемо:**
- Последний save wins (стандартное поведение)
- Нет конфликтов/крашей

## Примечания

1. **Auth gating:** Все запросы должны стартовать только после `!authLoading && token && isAuthenticated`
2. **Обработка ошибок:** 409/404/403 обрабатываются с правильными UI блоками (warning/error)
3. **Conditions для complex:** Отправляется массив (как в WEB), даже если backend ожидает dict (SQLAlchemy JSON принимает оба)
4. **Refetch после операций:** После create/update/delete всегда вызывается `loadDiscounts()` для обновления списков
