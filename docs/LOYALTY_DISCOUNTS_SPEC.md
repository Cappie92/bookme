## Сущности и данные

### 1. LoyaltyDiscount (быстрые и сложные скидки)

**Таблица:** `loyalty_discounts`  
**Модель:** `backend/models.py:1042-1082`

**Поля:**
- `id` (int, PK)
- `salon_id` (int, FK → salons.id, **обязательно**)
- `discount_type` (enum: `'quick'` | `'complex'`)
- `name` (string) — название скидки
- `description` (text, nullable) — описание
- `discount_percent` (float, 0-100) — процент скидки
- `max_discount_amount` (float, nullable) — максимальная сумма скидки в рублях (null = без ограничений)
- `conditions` (JSON) — условия применения (структура зависит от типа)
- `is_active` (bool, default=True) — активна ли скидка
- `priority` (int, 1-10, default=1) — приоритет (для выбора максимальной скидки)
- `created_at`, `updated_at` (datetime)

**Индексы:**
- `idx_loyalty_discount_salon` (salon_id)
- `idx_loyalty_discount_type` (discount_type)
- `idx_loyalty_discount_active` (is_active)

**Ограничения:**
- `discount_percent`: 0-100 (валидация в схеме)
- `priority`: 1-10 (валидация в схеме)
- Привязка к `salon_id` (не к `master_id`)

### 2. PersonalDiscount (персональные скидки)

**Таблица:** `personal_discounts`  
**Модель:** `backend/models.py:1085-1116`

**Поля:**
- `id` (int, PK)
- `salon_id` (int, FK → salons.id, **обязательно**)
- `client_phone` (string) — номер телефона клиента (pattern: `^\+?1?\d{9,15}$`)
- `discount_percent` (float, 0-100) — процент скидки
- `max_discount_amount` (float, nullable) — максимальная сумма скидки в рублях
- `description` (text, nullable) — описание
- `is_active` (bool, default=True) — активна ли скидка
- `created_at`, `updated_at` (datetime)

**Индексы:**
- `idx_personal_discount_salon` (salon_id)
- `idx_personal_discount_phone` (client_phone)
- `idx_personal_discount_active` (is_active)

**Ограничения:**
- `client_phone` должен соответствовать существующему пользователю (проверка при создании)
- Уникальность: один клиент (по телефону) может иметь только одну персональную скидку в салоне

### 3. AppliedDiscount (примененные скидки)

**Таблица:** `applied_discounts`  
**Модель:** `backend/models.py:1119-1144`

**Поля:**
- `id` (int, PK)
- `booking_id` (int, FK → bookings.id, **обязательно**)
- `discount_id` (int, FK → loyalty_discounts.id, nullable) — для quick/complex
- `personal_discount_id` (int, FK → personal_discounts.id, nullable) — для personal
- `discount_percent` (float) — примененный процент
- `discount_amount` (float) — примененная сумма в рублях
- `applied_at` (datetime) — когда применена

**Назначение:** История применения скидок (для аналитики и отчетов)

**⚠️ ВАЖНО:** В текущей реализации скидки **НЕ применяются автоматически** при создании бронирования. Таблица `applied_discounts` существует, но логика применения не реализована (см. раздел "Применение скидок").

### 4. QuickDiscountTemplate (шаблоны быстрых скидок)

**Источник:** Константа `QUICK_DISCOUNT_TEMPLATES` в `backend/routers/loyalty.py:34-117`

**Шаблоны:**
1. **"Новый клиент"** (`first_visit`)
   - Условие: первая запись клиента
   - Дефолтная скидка: 10%

2. **"Регулярные визиты"** (`regular_visits`)
   - Условие: N визитов за период (по умолчанию: 5 визитов за месяц)
   - Дефолтная скидка: 15%

3. **"Возвращение клиента"** (`returning_client`)
   - Условие: клиент не был N дней (по умолчанию: 30 дней)
   - Дефолтная скидка: 20%

4. **"День рождения"** (`birthday`)
   - Условие: дата записи в диапазоне дней до/после дня рождения (по умолчанию: ±7 дней)
   - Дефолтная скидка: 25%

5. **"Счастливые часы"** (`happy_hours`)
   - Условие: время записи в диапазоне (по умолчанию: 09:00-12:00, пн-пт)
   - Дефолтная скидка: 15%

6. **"Скидка на услуги"** (`service_discount`)
   - Условие: определенные услуги или категории услуг
   - Дефолтная скидка: 10%

**Структура шаблона:**
```python
{
    "id": "first_visit",
    "name": "Новый клиент",
    "description": "Скидка за первую запись",
    "icon": "🎁",
    "conditions": {
        "condition_type": "first_visit",
        "parameters": {}
    },
    "default_discount": 10.0
}
```

## Сценарии пользователя (Web UI)

### Вкладка "Быстрые скидки"

**Компонент:** `LoyaltySystem.jsx` → `QuickDiscountsTab`

**Что отображается:**
1. **Сетка шаблонов** (6 шаблонов из `QUICK_DISCOUNT_TEMPLATES`)
   - Иконка, название, описание
   - Дефолтный процент скидки
   - Бейдж "Активна" если скидка уже создана
   - Кнопка "Активировать" или "Активна" (disabled)

2. **Список активных быстрых скидок**
   - Название, описание, процент
   - Кнопка редактирования процента (карандаш)
   - Кнопка удаления (корзина)

**Действия:**
- **Активировать шаблон:**
  1. Пользователь нажимает "Активировать" на шаблоне
  2. Frontend вызывает `POST /api/loyalty/quick-discounts` с данными шаблона
  3. Backend создает `LoyaltyDiscount` с `discount_type='quick'`
  4. Frontend обновляет список (`loadData()`)

- **Редактировать процент:**
  1. Пользователь нажимает карандаш на активной скидке
  2. Вводит новый процент (0-100)
  3. Frontend вызывает `PUT /api/loyalty/quick-discounts/{id}` с `{discount_percent: newValue}`
  4. Backend обновляет скидку
  5. Frontend обновляет список

- **Удалить скидку:**
  1. Пользователь нажимает корзину
  2. Подтверждение через `confirm()`
  3. Frontend вызывает `DELETE /api/loyalty/quick-discounts/{id}`
  4. Backend удаляет скидку
  5. Frontend обновляет список

**Валидация (frontend):**
- Процент: 0-100, step 0.1
- Проверка на активность (нельзя активировать дважды)

**Валидация (backend):**
- `discount_percent`: 0-100 (Pydantic `Field(ge=0, le=100)`)
- `priority`: 1-10 (Pydantic `Field(ge=1, le=10)`)

## Правила применения скидок

### ⚠️ КРИТИЧЕСКИЙ МОМЕНТ

**В текущей реализации скидки НЕ применяются автоматически при создании бронирования.**

**Что есть:**
- Таблица `applied_discounts` для истории
- Эндпоинт `GET /api/loyalty/check-discount/{client_phone}` для проверки (но проверяет только персональную скидку)
- Модели и схемы готовы

**Чего нет:**
- Логики автоматического применения скидок при создании бронирования
- Логики проверки условий (first_visit, regular_visits, birthday, happy_hours и т.д.)
- Логики выбора максимальной скидки при нескольких применимых
- Логики расчета финальной цены с учетом скидки

**Где должна быть логика:**
- При создании бронирования (`POST /api/client/bookings` или `POST /api/bookings`)
- При расчете стоимости услуги
- При подтверждении бронирования (`POST /api/master/accounting/confirm-booking/{id}`)

**Текущее состояние:**
- Скидки создаются и управляются через UI
- Но не применяются автоматически
- Таблица `applied_discounts` не заполняется

---

### Предполагаемые правила (на основе структуры данных)

**Приоритеты:**
1. **Персональная скидка** — наивысший приоритет (если есть для клиента)
2. **Сложные скидки** — проверяются по условиям, выбирается максимальная по `priority`
3. **Быстрые скидки** — проверяются по условиям, выбирается максимальная по `priority`

**Выбор скидки:**
- Если несколько скидок применимы — выбирается с максимальным `priority`
- Если `priority` одинаковый — выбирается максимальный `discount_percent`
- Если `discount_percent` одинаковый — выбирается первая найденная

**Ограничения:**
- `max_discount_amount` — если задан, скидка не может превышать эту сумму
- `is_active` — только активные скидки применяются
- Условия в `conditions` (JSON) — должны выполняться для применения

**Условия (из шаблонов):**
- `first_visit` — первая запись клиента у салона
- `regular_visits` — N визитов за период (нужно считать из `Booking` где `status='completed'`)
- `returning_client` — дней с последнего визита >= N
- `birthday` — дата записи в диапазоне дней до/после дня рождения клиента
- `happy_hours` — время записи в диапазоне часов и дней недели
- `service_discount` — услуга входит в список `service_ids` или категория в `category_ids`
